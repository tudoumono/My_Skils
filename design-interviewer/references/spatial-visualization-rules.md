# spatial-visualization-rules.md — 空間・地図・立体可視化ルール

## 概要

Spatial UI / Map UI / 展示・イベント・AR/ナビゲーション文脈で使う。奥行き・レイヤー・地図・3Dは、位置・導線・関係性が価値になる時だけ理解を助ける。装飾で使うと酔わせるだけになる。「空間表現は情報構造に一致させる」が鉄則。

## 向いている用途

- 空間コンピューティング/AR、来場者ナビ、施設・店舗案内、物流・位置ベースサービス。
- 展示ブース・サイネージなど、体験と空間が主役の場。
- ワークフロー・依存関係・AI処理の流れなど、関係性そのものを見せる可視化。

## 向いていない用途

- 素早く読ませたい文書、機能比較、高密度の業務ダッシュボード、長文中心の説明。
- 位置・空間・関係性が価値に関係しない抽象サービス（無理に3D/地図を足さない）。
- 低スペック環境で常用される業務UI。

## 判断基準（空間表現の原則）

- **奥行き＝情報階層**: 手前ほど重要、奥ほど補助。奥行きが優先度と一致していないなら、その3Dは装飾。
- **2Dで十分なら3D化しない**: 2Dでも伝わる情報を無理に立体化しない。空間表現は理解・操作を助ける時だけ。
- **操作対象と装飾を区別**: 何がクリック/タップできるかが見た目で分かること。
- **地図は目的の情報だけ強調**: 地図上のピン/ルート/エリアは、見せたい情報に絞る。全部を光らせない。
- **動きは意味を持たせる**: 視差・アニメは「注目を誘導する」「関係を示す」目的に限定。装飾の常時アニメは酔わせる。
- **環境で破綻させない**: モバイル・低解像度・低性能端末で成立するか。アクセシビリティ上の代替（2D/テキスト）があるか。
- **視認距離を想定**: 展示・サイネージは遠くから読める最小サイズを大きく取る（`theme-presets.md` ④）。

## 推奨パターン

- レイヤー（前景/中景/背景）に情報の役割を割り当てる。前景=主メッセージ、背景=文脈。例: 現在地、候補、混雑、導線。
- 地図UIは、目的の場所/ルートをAccent、周辺はグレーに退かせる（`data-viz-rules.md` の主役1色の考え方と同じ）。地図を背景ではなく意思決定材料にする。
- 奥行きは影・サイズ・重なり・ぼかしを控えめに使って表す。
- 3D/空間演出は導入・体験の"見せ場"に限定し、可読領域と重要な操作は平面・明基調のUIに戻す。
- ナビゲーションは「今どこ・次どこ・あと何歩/何分」を明快に。
- 3D表現を使う時は、読み込み・低性能端末・静止代替を想定し、2Dリストやテキスト案内のフォールバックを用意する。

## NGパターン

- 意味のない浮遊カード・回転オブジェクトで奥行きを装飾に使う。
- 操作できそうでできない浮遊カード（操作対象と装飾の混同）。
- 地図を背景画像として貼るだけで、位置情報が機能していない。
- 常時動く視差で、内容を読もうとすると酔う。
- ピンを多色で乱立させ、どれが目的地か分からない。
- 遠距離視認が必要なのに文字が小さく、現地で読めない。
- 2Dで十分な説明を3Dで複雑化する。

## レビュー質問

1. 奥行き/レイヤーは情報の重要度・関係性と一致しているか。装飾になっていないか。
2. 地図上の強調は目的の情報に絞れているか。周辺は退いているか。地図は意思決定材料になっているか。
3. ユーザーはどこを見て・どこを操作すべきか分かるか。重要な操作は平面UIに残っているか。
4. 動き/アニメに意味（誘導・関係提示）があるか。装飾の常時アニメになっていないか。
5. 地図や3Dが表示できない環境でも最低限伝わるか（2D/テキストの代替）。モバイル・低性能端末で破綻しないか。
6. 展示なら、想定視認距離で読めるサイズか。実装コストに見合う効果があるか。

## 出力例

> 展示ブースの来場者ナビはMap UI主体。地図中央に「今いる場所」を大きく、目的のブースへのルートをAccentで1本だけ強調し、他の通路・施設はグレーに退かせます。奥行きは"現在地レイヤー（前景）＞ルート＞混雑度＞背景地図"で情報階層に一致。演出アニメは到着時の1回だけ。文字は3m先から読める大きさを最小基準にし、詳細情報はタップで平面パネルに展開。低性能端末向けに2Dリスト表示とテキスト案内のフォールバックを用意します。
