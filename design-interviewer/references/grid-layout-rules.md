# grid-layout-rules.md — グリッド・レイアウトルール

## 概要

Phase 9 で使う。グリッドは装飾ではなく「構造」。重要度を色に頼らず、サイズ・位置・余白・密度で作るための基準をまとめる。レイアウト・余白の話になったら読む。

## 中心思想

- グリッドは"揃えるため"ではなく"情報の階層を作るため"に使う。
- 重要なものほど「大きく・上に・余白を伴って・独立して」置く。
- 余白は不足でなく設計。余白の大きさに一貫した基準（8px系）を持たせる。

## 扱う要素

12カラムグリッド / 8pxベースライン / ガター / マージン / max-width / 左揃え / 余白による整理 / Bentoのカードサイズ差 / ダッシュボードの密度 / 資料のモジュラーグリッド。

## 向いている用途 / 向いていない用途

- 向いている: LP構成 / Webアプリ画面 / ダッシュボード / スライド資料 / インフォグラフィック。
- 向いていない: グリッドを見た目の飾りとして使うこと。コンテンツ量を無視してカードに詰め込むこと。

## 12カラムグリッド

- PCは12カラムを基本にする。2/3/4/6分割が作れて柔軟。
- 主コンテンツは 8カラム、補助（サイドバー等）は 4カラム、のような役割配分をする。
- カラムをまたぐ要素の左右端を、他要素の基準線に合わせる。

## 8pxベースライン（スペーシング）

- 余白・間隔は **8の倍数**を基本にする（4pxは微調整用）。例: 8 / 16 / 24 / 32 / 48 / 64。
- 要素間の余白に意味を持たせる: 関連が強いものは近く（8〜16）、区切りは大きく（32〜64）。
- 「なんとなく20px」のような基準外の値を散らさない。

## ガターとマージン

- ガター（カラム間の間隔）は一定に保つ（目安: 24〜32px）。バラつくと雑に見える。
- ページ左右のマージンは十分に取り、内容を端に貼り付けない。
- カード内パディングも8px系で統一する。

## max-width

- 本文・コンテンツ幅に上限を設ける（長文が横に伸びると読めない。`typography-guide.md` の行長と連動）。LPなら 1120〜1280px が目安。
- 全幅（フルブリード）は写真・区切り帯・Hero背景など、意図した箇所だけに使う。

## 左揃え

- 本文・箇条書き・見出しは基本 **左揃え**。読み始めの位置が揃い、視線が安定する。
- 中央揃えは「表紙」「Heroの短いコピー」「短い引用」など短文に限定する。長文を中央揃えにしない。

## 余白による情報整理

- 線や色で区切る前に、まず余白で区切れないか考える。余白は最も静かで強い区切り。
- グルーピングは「近接」で表す。関連する要素同士を近づけ、無関係なものを離す。

## Bento UI のカードサイズ差

- カードの大小は**重要度**を表す。主役の機能/指標を大きいカードに。
- 全カード同サイズは避ける（優先順位が消える）。ただしサイズ段階は2〜3種に留める（増やすと散らかる）。
- カードのガター・角丸・パディングは全カードで統一する。
- 実データを入れてもカードが伸びて崩れないか、コンテンツ量を想定して設計する。

## 用途別の数値目安

### LP

- 12カラム / max-width 1120〜1280px / ガター 24〜32px。
- セクションの上下パディングは 64〜120px。余白でセクションを区切る。
- Heroは左揃え基本。中央揃えは短いコピーや表紙的表現のみ。

### Webアプリ

- サイドバー: 240〜280px / ヘッダー: 56〜72px / 詳細パネル: 320〜420px を目安。
- メインエリアは可変幅。主要操作は上部または右上に固定する。
- 余白は 8 / 12 / 16 / 24 / 32px を中心に使う。

## ダッシュボードの情報密度

- 最重要KPIを最上位・大きく。二次情報は下または小さく。KPIカード上段 → 時系列・比較の中段 → 明細テーブル下段、の順に深掘りさせる。
- フィルタは上部または左側にまとめる。
- 密度は上げてよいが、行間・区切り・余白で「読める密度」を保つ。見出し・数値・単位・期間は分けて示す。
- アラートは色だけでなくアイコン・文言・位置でも示す。
- グラフ・テーブル・カードを同じ基準線に乗せる。`data-viz-rules.md` と連動。

## 資料（スライド）のモジュラーグリッド

- 16:9 を前提に、スライドも見えないグリッドに乗せる。タイトル位置・本文開始位置・余白を全ページで統一。
- 1スライド1メッセージ。要素を詰め込まない。
- 左に主張、右に図解や証拠を置くと読みやすい。
- 図・箇条書き・キャプションの左端を揃える。表紙・章扉以外で中央揃えを多用しない。

## 必須ルール

1. グリッドは構造として使う（装飾で崩さない）。
2. 重要度は色でなく、サイズ・位置・余白・密度で表現する。
3. 余白は8px系の倍数を基本にする。
4. 中央揃えは表紙・短いコピーに限定する。
5. 見出し・本文・画像・カードを同じ基準線に乗せる。
6. スマホでは縦積みの順序を設計する（重要なものから）。順序が破綻しないか確認する。

## NGパターン

- 余白の値がバラバラ（20/18/26…）で、要素が微妙にずれて見える。
- 全カード同サイズで、どれが主役か分からない。
- 長い本文が中央揃え＆全幅で、読みにくい。
- 重要度を色の派手さだけで表現し、レイアウトは平坦。
- 要素の左端が揃っておらず、ガタついて見える。
- 実データを入れるとカードが伸びてレイアウトが崩れる。

## レビュー質問

1. 最初に見てほしいものはどこか。重要な情報が自然な視線順に並んでいるか。
2. すべての余白は8px系か。基準外の値が紛れていないか。
3. 最重要要素は「大きく・上に・余白を伴って」いるか。余白を減らしても意味の階層が壊れないか。
4. 主要な要素の左端は揃っているか。
5. 中央揃えは短文だけに限定されているか。
6. 色を消しても、レイアウトだけで優先順位が伝わるか。
7. （Bento）カードの大小が重要度を表しているか。サイズ段階は2〜3種に収まっているか。
8. スマホで読む順番は自然か。空状態・エラー状態でもレイアウトが保たれるか。

## 出力例

> レイアウトは12カラム・8pxベースライン・ガター24pxで組みます。Heroは全幅、以降のコンテンツは max-width 1200px 内に左揃えで配置（左6カラムにコピー、右6カラムにプロダクト画面）。機能はBentoで、主役機能を2カラム幅の大カード、他を1カラムの小カードにして重要度を示します。余白は関連16px/区切り48px/セクション96pxで階層を作り、色に頼らず優先順位を表現。スマホでは Hero → CTA → 主要ベネフィット → 証拠 の順に縦積みします。
