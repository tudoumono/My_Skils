# question-bank.md — 質問バンク

## 概要

Phase 0〜3で使う質問の在庫。質問はデザインを作るためではなく、**判断を減らすため**に使う。すべて「選択肢 + その他/自由回答」で答えられる形にしてある。AIはこのまま丸ごと投げず、状況に応じて**1ターン最大3問**を選んで使う。ユーザーが答えたら「ここまでの理解」で要約し、ズレを潰してから次へ進む。

## 使い方の判断基準

- 引数がほぼ空 → Phase 0 起点3問から。
- 引数が具体的 → 埋まった項目は「仮の理解」として提示し、不足分だけ聞く。
- ユーザーが即答できない質問は、選択肢を先に読み上げて負荷を下げる。
- 「わからない」も有効な回答。仮説を置いて未決事項に記録し、前進する。
- 質問を選ぶ基準: その答えで次の判断が進むか / 専門用語なしで答えられるか / 選択肢が仮説の提示になっているか / 自由回答で例外を受け止められるか。
- 聞いてはいけないもの: すでに合意済みの内容の再確認、完成を遅らせるためだけの質問。

## Phase 0: 起点3問（最初の応答で必ず使う）

**Q1. 作りたいものはどれに近いですか？**
- A. LP（サービス紹介の1枚ページ）
- B. アプリ画面（Web/モバイルの操作画面）
- C. ダッシュボード / 分析画面
- D. スライド資料（提案・社内説明など）
- E. インフォグラフィック（1枚で伝える図解）
- F. その他（管理画面・ブランド案・サービス紹介ページ 等）→ 自由記述

**Q2. 誰に見せるものですか？**
- 一般ユーザー / 社内 / 顧客 / 投資家 / 採用候補者 / 技術者 / その他（自由記述）

**Q3. 見た人に最終的に何をしてほしいですか？（最終CTA）**
- 問い合わせ / 登録 / 購入 / 理解（腹落ち） / 導入判断 / 社内承認 / その他（自由記述）

> 出力例（最初の応答）:
> 「会議アシスタントアプリのLPですね。方向性を外さないために、まず3つだけ教えてください。（Q1〜Q3を提示）。ざっくりで大丈夫です。決まっていなければ『わからない』でも構いません。」

## Phase 1: 目的・ターゲット整理

聞き出す対象: サービス名 / 概要 / 誰向けか / ユーザーの悩み / 利用シーン / 伝えたい価値 / 競合・類似サービス / 取ってほしい行動。

**Q. サービスを一言でいうと？（穴埋めで）**
- 「〔誰〕が〔どんな状況〕で〔何〕できるようになる、〔種類〕のサービス」を埋めてください。

**Q. 主なターゲットは？（複数可）**
- 現場担当者 / 管理職・決裁者 / 経営層 / エンドユーザー個人 / 情シス・技術者 / その他

**Q. そのターゲットが今いちばん困っていることは？**
- 時間がかかる / ミスが多い / 属人化している / 情報・判断材料が散らばって探せない / 成果が見えにくい / コストが高い / その他（自由記述）

**Q. 何を一番伝えたいですか？（1つに絞る）**
- 便利さ / 安心感 / 導入効果 / 新しさ / 専門性 / 価格の納得感 / その他（自由記述）

**Q. どんな場面で見られますか？**
- PCでじっくり / スマホで移動中 / 会議のスクリーン投影 / 展示ブースで立ち止まって / 提案の場で1対1 / その他

**Q. 似ているサービスや「これに近い雰囲気」はありますか？**
- ある（名前や印象を自由記述） / ない / 逆に「これは避けたい」もあれば

> レビュー質問（自分に問う）: ターゲットは1種類に絞れているか。複数なら主・従を決めたか。悩みは"機能"でなく"痛み"の言葉になっているか。伝えたい価値は1つに絞れたか。

## Phase 2: 情報設計

聞き出す対象: 最初の3秒で伝えたいこと / 不安点 / 機能説明の順番 / 信頼材料 / 実績 / 料金の有無 / 導入ステップ / FAQ / CTA。

**Q. 最初の3秒で伝わってほしい一言は？**
- 自由記述（例: 「議事録をゼロにする」）。無ければ「誰のための何か / 何が楽になるか / どんな成果が出るか / なぜ信頼できるか」のどの軸で言うかを選んでもらい、AIが仮案を3つ出して選んでもらう。

**Q. 見た人が不安・疑問に思いそうなことは？（複数可）**
- 本当に効果あるの / 難しそう・使いこなせるか / 高そう / セキュリティは / 導入が大変そう / 自社に合うか / 他社と何が違うの / その他

**Q. 信頼材料として出せるものは？**
- 導入社数・実績 / 顧客ロゴ / 数値効果 / 受賞・認証 / 開発体制・専門性 / まだ無い（→未決事項）

**Q. 料金・導入ステップの情報は載せますか？**
- 料金を明記 / 「お問い合わせ」に寄せる / 無料トライアルあり / 導入ステップを見せる / まだ未定

> レビュー質問: 「最初の3秒」→「不安の解消」→「行動」の順で情報が流れる設計になっているか。信頼材料が無い場合、代わりに何で信頼を作るか決めたか。

### ビジュアル方向性（Phase 2〜3の間で、3案提示の前に確認）

**Q. 近い印象はどれですか？（1〜2個）**
- 信頼できる / 軽快で使いやすい / 先進的 / 高級感がある / 親しみやすい / 記憶に残る / その他（自由記述）

**Q. 避けたい印象はどれですか？（1〜2個）**
- 安っぽい / 子どもっぽい / 冷たい / 固すぎる / 派手すぎる / 既視感が強い / その他（自由記述）

> 「近い印象」と「避けたい印象」の組み合わせが、後の3案（Safe/Attractive/Distinctive）の振れ幅を決める。

## Phase 3: 構成設計（成果物タイプ別の骨格を"提案"する）

ここではユーザーに一から考えさせず、**タイプ別の標準骨格を提示して、要否と順序を調整してもらう**。各セクションに「何を載せるか」を一言添える。

### LPの標準骨格
Header / Hero（3秒メッセージ+CTA） / Problem（悩みの言語化） / Solution（それをどう解くか） / Features（主要機能3〜5） / Use Case（利用シーン） / How it works（導入の流れ） / Trust・Security（信頼材料） / Pricing / FAQ / CTA（再掲） / Footer。

- 調整質問: 「この中で不要なセクション、逆に足したいものはありますか？ 順番の希望は？」

### アプリ画面の標準骨格（状態設計を含む）
Header / Sidebar（ナビ） / Main Area（主コンテンツ） / Detail Panel（詳細） / Action Area（操作） / Empty State（データ0件） / Loading State / Error State。

- 調整質問: 「1画面ですか、複数画面ですか？ 一番使う操作は何ですか？ データが0件のときは何を見せますか？」

### スライド資料の標準骨格
表紙 / 背景（なぜ今） / 課題 / 解決策 / 仕組み / 導入効果 / 実装計画 / 判断ポイント / まとめ（次アクション）。

- 調整質問: 「意思決定の場ですか、理解の場ですか？ 決裁者は何が揃えばYesと言えますか？」

### 管理画面・ダッシュボードの標準骨格
Header / フィルタ・期間選択 / KPIサマリ（重要指標3〜5） / 主グラフ（時系列/比較） / 明細テーブル / ドリルダウン詳細 / アラート・通知（必要なら） / Empty・Loading・Error。

- 調整質問: 「毎日見て何を判断しますか？ 最重要の1指標は？ 期間やフィルタは何が必要ですか？（`data-viz-rules.md` を参照）」

### インフォグラフィックの標準骨格
つかみの一言 / 前提となる数字 / 対比または流れ / 結論となる1メッセージ / 出典。

- 調整質問: 「1枚で1つのことだけ言うとしたら、それは何ですか？」

### 最終成果物の形式確認（Phase 3の締めで1問）

**Q. 最後に何を出しますか？**
- LP構成（テキスト） / HTMLモック / React画面案 / スライド構成 / デザイントークン / ブランド案 / その他（自由記述）

> レビュー質問: 標準骨格をそのまま押し付けていないか。ユーザーの事情（信頼材料の有無・画面数・意思決定の性質）で削る/足すの判断ができているか。成果物の形式まで合意したか。

## 曖昧回答の受け止め方（共通テクニック）

- 「おしゃれにして」→ 「おしゃれ」を分解して確認する: 余白多め/写真主役/色少なめ/上質な字組み、のどれに近いか選択肢化する。
- 「かっこよく」→ Safe/Attractive/Distinctive のどの方向かに翻訳し、後の3案に接続する。
- 「AIっぽく」→ そのまま青紫グラデに行かず、「先進的＝明基調で余白と精度で見せる」「近未来＝ダークUI+限定的な発光」「実データや処理フローでAIらしさを見せる」のどれかを選ばせる（`anti-ai-slop-rules.md`）。
- 「他社みたいに」→ 具体名を1つ引き出し、その"どこ"が良いのか（色/情報量/写真や図解の使い方/コピーの声色）を特定する。

## NGパターン（やってはいけない聞き方）

- 「トンマナはどうしますか？」「好きな色は？」から始める。
- 「参考サイトをください」で止まる。
- 全フェーズの質問を一度に並べる。
- 不明点があるたびに作業を止める（仮説を置いて進む）。
- 「おしゃれ」「かっこいい」「AIっぽい」を翻訳せず、そのまま見た目に反映する。

## 出力例（Phase末の「ここまでの理解」）

> ここまでの理解:
> - 成果物: LP / ターゲット: 中小企業の情シス担当（主）と経営層（従）
> - 痛み: 議事録作成に毎回30分かかり属人化している
> - 3秒メッセージ: 「会議のあと、議事録はもう書かない」
> - 最終CTA: 無料トライアル登録
> - 仮説: 信頼感を最優先しつつ、余白と図解で見栄えを作る
> - 未決: 料金の明記可否、導入実績の数値
> この理解で Phase 3（構成）に進みます。ズレていれば直してください。
